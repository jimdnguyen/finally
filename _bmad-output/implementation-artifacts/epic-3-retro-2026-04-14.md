**3-2 — Chat Panel UI**
- JSDOM `scrollIntoView`: `bottomRef.current?.scrollIntoView?.()` double optional chain required — JSDOM does not define `HTMLElement.prototype.scrollIntoView`; single `?.` on `current` still throws when calling the undefined function.

- 3 pre-existing test failures confirmed via `git stash` — not introduced by 3.2.

**3-3 — AI Error Handling & Mock Mode**
- Custom `@app.exception_handler(HTTPException)` in `app/main.py` bypasses FastAPI's default `{"detail": ...}` envelope — response body IS `exc.detail` directly. Router-level test initially asserted `data["detail"]["code"]` (wrong); corrected to `data["code"]`.
- Discriminated union optional field as affordance gate: `retryText?: string` on `exec-fail` variant only — presence = Retry button shown, value = payload for re-submission. Cleaner than a boolean flag.
- Code review result: **0 decision_needed · 0 patch · 4 defer · 6 dismiss** — cleanest review of the project.

## What Went Well

- **Schema-first design**: Defining the structured output format (`{message, trades, watchlist_changes}`) before writing service code meant no mid-story pivots. Every downstream decision was grounded.
- **Mock mode architecture**: `LLM_MOCK=true` was baked in from 3.1 design. Story 3.3 testing was clean — no hacks, no special doubles, just flip the env var.
- **Discriminated union extensibility**: Adding `retryText?: string` to one variant in 3.3 was surgical — zero blast radius on other cases.
- **Split test file pattern**: Four test files per domain (models/db/service/api) — fast, targeted, maintainable. Easier to locate failures.
- **300 tests, 0 regressions**: Across all three stories, including a new async service layer, LiteLLM integration, and custom exception handling.
- **3.3 code review**: Cleanest in the project — 0 patch items, 0 decisions needed.

## What Was Hard

- **Custom exception handler invisibility**: `@app.exception_handler(HTTPException)` in `main.py` changes the response envelope globally. Not obvious when writing a new router-level test. The rule going forward: always check `main.py` for custom exception handlers before writing router-level error assertions.
- **Cross-story dependency gap**: `portfolioStore.refresh()` was specified in 2.1 but not implemented. Story 3.2 had to add it mid-story. Not a catastrophe — documented and fixed — but a pattern to watch.
- **Open patch items carrying forward**: 4 items from Story 3.1 review and the `watchlist_changes_applied` status mismatch (`'added'` vs `'ok'`) are live issues. Real probability of production impact on the fetchone crash and the contract mismatch.

## Patterns Identified

1. **Custom exception handler response shape**: Application-level `@app.exception_handler` overrides FastAPI defaults globally. Router-level integration tests must check `main.py` before asserting error response shape. Service tests and router tests can give different results for the same error code.

2. **Optional variant fields as affordance gates**: When a UI affordance (button, action) only makes sense on some list entries, prefer `optionalField?: PayloadType` over `canDoThing: boolean`. Presence = enabled. Value = the data needed to perform the action. The field is simultaneously gate and payload.

3. **Fix-document-move-on**: Cross-story dependency gaps and pre-existing issues are best handled by fixing them in the current story, documenting the fix in the Dev Agent Record, and continuing — not by blocking or escalating. The key is documentation so future context is preserved.
