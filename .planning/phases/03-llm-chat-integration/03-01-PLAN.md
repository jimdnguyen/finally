<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User Input → Chat Endpoint | User message from `/api/chat` request body |
| LLM Response → Validation | JSON returned from OpenRouter via LiteLLM |
| Validated Schema → Trade Execution | ChatResponse trades field used by service to execute trades |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | LLM JSON response | Mitigate | Pydantic v2 .model_validate_json() validates response against ChatResponse schema before service code sees it; malformed JSON raises ValidationError |
| T-03-02 | Tampering | TradeAction fields | Mitigate | Field constraints: side in (buy, sell), quantity > 0, ticker 1-5 chars; enforced at deserialization time by Pydantic |
| T-03-03 | Information Disclosure | Chat history in DB | Accept | Chat messages stored in SQLite (not encrypted); CHAT-05 persists user messages + executed actions; single-user demo, no PII, low risk |
| T-03-04 | Elevation of Privilege | LLM auto-execution | Accept | LLM can execute trades without explicit user confirmation (PLAN.md §9 design); simulated environment with fake money, intentional feature for demo value |

</threat_model>
