Add async function `execute_trade()` to backend/app/portfolio/service.py. This function orchestrates the entire atomic trade execution.

**Function signature:**
```python
async def execute_trade(
    db: sqlite3.Connection,
    ticker: str,
    side: str,
    quantity: Decimal,
    price_cache: PriceCache,
) -> dict:
```

**Behavior (must follow this exactly):**

1. **Pre-validate** (no lock held):
   - Call `validate_trade_setup(db, ticker, side, quantity, price_cache)` 
   - If invalid: raise `HTTPException(status_code=400, detail=error_msg)`

2. **Fetch current price** from cache:
   - `price_update = price_cache.get(ticker)`
   - If None: raise `HTTPException(status_code=400, detail=f"No price for {ticker}")`
   - Convert to Decimal: `current_price = Decimal(str(price_update.price))`

3. **Wrap entire trade flow in `BEGIN IMMEDIATE` transaction**:
   - Acquire write lock early to prevent phantom reads
   - Inside transaction, re-validate (in case price data changed) before proceeding
   - If validation fails: raise HTTPException, rollback will occur automatically

4. **Execute trade inside transaction:**
   
   a) **Fetch current state** (fresh from DB, now locked):
      - Cash balance from users_profile WHERE id='default'
      - Current position (if exists) from positions WHERE user_id='default' AND ticker=?
   
   b) **Validate inside transaction** (redundant check for safety):
      - For buy: `cost = quantity × current_price`, check `cost <= cash_balance`
      - For sell: check `quantity <= current_quantity`
      - On failure: raise HTTPException
   
   c) **Execute buy:**
      - Calculate: `new_cash = cash_balance - (quantity × current_price)`
      - Update users_profile: SET cash_balance=new_cash WHERE id='default'
      - If position exists: UPDATE positions SET quantity=new_qty, avg_cost=new_avg_cost (using weighted average formula from RESEARCH.md Pattern 4)
      - If position doesn't exist: INSERT new position with quantity, avg_cost=current_price
   
   d) **Execute sell:**
      - Calculate: `new_cash = cash_balance + (quantity × current_price)`
      - Update users_profile: SET cash_balance=new_cash WHERE id='default'
      - Calculate: `new_qty = current_quantity - quantity`
      - If new_qty > 0: UPDATE positions SET quantity=new_qty
      - If new_qty == 0: DELETE FROM positions (sell-to-zero case, per Pitfall 5)
   
   e) **Record trade log entry** (immutable audit trail):
      - INSERT INTO trades (id, user_id, ticker, side, quantity, price, executed_at)
      - Use uuid.uuid4() for id, 'default' for user_id, datetime('now') for executed_at
   
   f) **Record portfolio snapshot** (immediately post-trade, same transaction):
      - Call compute_portfolio_value(cursor, price_cache) to get total value
      - INSERT INTO portfolio_snapshots (id, user_id, total_value, recorded_at)
      - Use uuid.uuid4() for id, 'default' for user_id, datetime('now') for recorded_at

5. **Commit transaction** (automatic on normal return, rollback on exception)

6. **Return dict with keys:**
   - `success: True`
   - `ticker: str`
   - `side: str`
   - `quantity: float(quantity_decimal)`
   - `price: float(current_price)`
   - `new_balance: float(new_cash)`
   - `executed_at: ISO timestamp string`

**Key requirements:**
- Use `run_in_threadpool(_execute_sync)` to wrap the entire sync transaction in async context
- All monetary values use Decimal, initialized from strings (never from floats)
- Weighted average cost calculation: `new_avg_cost = (current_qty × current_avg_cost + quantity × price) / (current_qty + quantity)`
- Use parameterized SQL queries (? placeholders) — never concatenate user input
- Store Decimal values in database as strings via `str(decimal_value)`
- Import: `from fastapi import HTTPException` (raise, don't return)

**Error handling:**
- HTTPException for validation failures (400 with descriptive detail)
- HTTPException for system errors (500)
- Let database exceptions bubble up (they will be caught by route handler try/except)

**No changes needed to:**
- `validate_trade_setup()` — reuse as-is from Phase 1
- `compute_portfolio_value()` — reuse as-is from Phase 1

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API | User input (ticker, side, quantity) crosses here; must validate via Pydantic before processing |
| API → Database | Trade execution requests cross here; must use parameterized queries to prevent SQL injection |
| Price Cache → Trade Execution | Live prices are fetched from cache; stale/invalid prices could cause incorrect trade validation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Tampering | TradeRequest validation | **mitigate** | Pydantic validates side ∈ {buy,sell}, quantity > 0, ticker format before reaching service layer; invalid requests rejected with HTTP 400 |
| T-02-01-02 | Tampering | SQL injection in trade execution | **mitigate** | All SQL queries use parameterized (?-placeholder) substitution; user input (ticker) never concatenated into SQL strings; Pattern 1 in RESEARCH.md verifies this |
| T-02-01-03 | Tampering | Race condition: validation → execution | **mitigate** | Entire trade flow wrapped in `BEGIN IMMEDIATE` transaction; write lock acquired before any state read; re-validation happens inside transaction; validates price hasn't changed dramatically |
| T-02-01-04 | Tampering | Float precision loss in P&L | **mitigate** | All monetary values use `Decimal` initialized from strings (never float); stored in DB as TEXT; no accumulation errors from IEEE 754 rounding |
| T-02-01-05 | Tampering | Insufficient funds check bypassed | **mitigate** | Buy validation checks `cost <= cash_balance` inside `BEGIN IMMEDIATE` transaction; re-validated if price changes; HTTPException raised before state change |
| T-02-01-06 | Tampering | Position quantity goes negative | **mitigate** | Sell validation checks `quantity <= current_qty` before execution; on sell-to-zero, position row is deleted (not zeroed); query filters `WHERE quantity > 0` |
| T-02-01-07 | Denial of Service | Unbounded trade quantity | **mitigate** | Pydantic Field validates quantity > 0 (lower bound enforced); upper bound implied by cash/shares owned; no limit-order queue, so max execution is instant |
| T-02-01-08 | Denial of Service | Snapshot recording blocks trades | **mitigate** | Snapshot is recorded inline during trade transaction (negligible cost); background task snapshots are separate and don't hold locks during trades |

**Summary:** All ASVS L1 mitigations in place. No unmitigated high-severity threats.
